# 오늘 뭐 먹지? 개발 플랜

## 1. 프로젝트 기본 셋업

- Next.js App Router + TypeScript 프로젝트 생성 (필요 시 `app/` 구조 초기화)
- Tailwind CSS 설치 및 기본 설정
- Supabase 프로젝트 생성 및 환경 변수(.env) 준비 (`SUPABASE_URL`, `SUPABASE_ANON_KEY` 등)
- `AGENTS.md`, `spec.md`, `api-spec.md`를 참고 규칙/스펙 문서로 고정

## 2. 데이터베이스(Supabase) 스키마 설계 및 생성

- `VoteRoom` 테이블 설계·생성 (id, title, team_id, expires_at, created_at 등)
- `MenuCandidate` 테이블 설계·생성 (id, room_id, name, description 등)
- `Vote` 테이블 설계·생성 (id, room_id, candidate_id, voter_id, created_at 등)
- `MealHistory` 테이블 설계·생성 (id, room_id, final_candidate_id, decided_at 등)
- 간단한 RLS/권한 정책 초안(초기에는 공개/테스트용 완화 정책도 가능)

## 3. 공용 Supabase 클라이언트 & 타입 정의

- 서버 컴포넌트/라우트에서 사용할 Supabase 클라이언트 헬퍼 파일 작성 (예: `lib/supabase/server.ts`)
- (선택) 클라이언트 컴포넌트용 Supabase 헬퍼 파일 작성
- `api-spec.md`의 개념 타입을 TypeScript 인터페이스/타입으로 정의 (예: `VoteRoom`, `MenuCandidate`, `Vote`, `MealHistory` 등)

## 4. 기본 UI 골격 만들기

- 메인 진입 페이지(`app/page.tsx`)에 간단한 랜딩 UI 구성
  - "투표 방 만들기" 버튼
  - "최근 메뉴 이력 보기" 링크
- 전역 레이아웃(`app/layout.tsx`)에서 기본 헤더/푸터·폰트·색상 정리

## 5. 투표 방 생성 플로우 구현

- `spec.md`와 `api-spec.md`의 **투표 방 생성** 요구사항을 기준으로 페이지 추가
  - `app/rooms/new/page.tsx`: 방 생성 폼 UI (제목, 팀 이름/ID(선택), 마감 시간)
- Supabase를 사용해 방을 생성하는 서버 액션 또는 `app/api/v1/rooms/route.ts` 구현
- 방 생성 성공 시 `/rooms/[roomId]`로 리다이렉트하는 로직 추가

## 6. 투표 방 상세 페이지 구현 (메뉴 제안 + 상태 조회)

- `app/rooms/[roomId]/page.tsx` 생성
- 서버 컴포넌트에서 Supabase를 통해 다음 데이터 조회
  - `VoteRoom` (방 정보)
  - 연결된 `MenuCandidate` 목록
  - 각 후보의 득표 수 집계 (JOIN 또는 별도 쿼리)
- UI 구성
  - 방 제목, 마감 시간 표시
  - 메뉴 후보 리스트 및 현재 득표 수 표시
  - 아래에 "메뉴 제안하기" 폼(입력창 + 추가 버튼)
- `POST /api/v1/rooms/:roomId/candidates`에 해당하는 서버 액션 또는 API 라우트 구현
  - 메뉴 후보를 Supabase에 INSERT
  - 성공 시 페이지 새로고침/상태 갱신

## 7. 1인 1표 투표 기능 구현

- 투표 방 상세 페이지에서 각 후보 카드/행에 "투표" 버튼 추가
- `POST /api/v1/rooms/:roomId/votes`에 해당하는 서버 액션 또는 API 라우트 구현
  - 같은 사용자가 같은 방에서 여러 번 투표하면 **기존 투표를 변경**하는 로직 설계
  - (초기 버전에서는 voterId 없이, 브라우저 세션/쿠키 기반 단순 제한도 허용)
- 투표 후 응답으로 전달된 집계 결과를 이용해
  - 후보별 득표 수 재계산 및 UI 갱신
  - 사용자가 어디에 투표했는지 표시(색상/아이콘 등)

## 8. 투표 결과 계산 및 표시

- `GET /api/v1/rooms/:roomId/result`에 해당하는 서버 로직 구현
  - 해당 방의 후보별 득표 수 집계
  - 최다 득표 메뉴 계산
  - 동점 시 tie-break 규칙 적용 (예: 먼저 제안된 메뉴 승리)
- 결과 화면 UI
  - "오늘의 메뉴: ○○○" 강조 표시
  - 득표 순위/막대그래프 스타일 리스트 (간단한 Tailwind 레이아웃)
- 결과 화면으로 이동하는 버튼/링크를 방 상세 페이지에 추가

## 9. 오늘의 메뉴 확정 및 식사 이력 저장

- 결과 확정 시 `MealHistory`에 기록하는 로직 구현
  - 서버 액션 또는 `POST /api/v1/rooms/:roomId/confirm` 라우트
  - `history` 레코드 생성 (roomId, finalCandidateId, decidedAt)
- 중복 확정 방지 (이미 이력이 있는 방이면 다시 생성하지 않도록 처리)

## 10. 최근 메뉴 이력 조회 페이지 구현

- `app/history/page.tsx` 또는 팀 기준 경로(`app/teams/[teamId]/history/page.tsx`) 생성
- `GET /api/v1/history?teamId=...&limit=...`에 해당하는 서버 로직 구현
  - 최근 N개의 `MealHistory`를 날짜 내림차순으로 조회
  - 각 기록에 연결된 메뉴 이름/방 제목도 함께 조회
- UI
  - 최근 날짜/메뉴 리스트로 단순하게 보여주기
  - 같은 메뉴가 자주 반복되는지 한눈에 보이도록 구성

## 11. 공유용 텍스트/링크 생성 기능 추가

- 결과 화면에서 "결정 내용 복사하기" 버튼 추가
  - 예: `오늘 디팀 점심 메뉴는 '초밥'으로 결정되었습니다! (링크)` 형식의 문자열 생성
  - 클립보드 복사 기능 구현(클라이언트 컴포넌트에서 처리)

## 12. 간단한 UX 다듬기

- 로딩/에러 상태 표시 (로딩 스피너, 에러 메시지 등)
- 폼 유효성 검사 (빈 제목, 빈 메뉴 이름 방지 등)
- 모바일 화면에서의 레이아웃·폰트 크기 점검

## 13. 추후 확장 아이디어 (선택)

- 사용자/팀 계정 시스템 도입 및 `teamId`, `voterId` 정식 사용
- 슬랙/카카오워크/디스코드 등 메시지 플랫폼과의 봇 연동
- 메뉴 통계(가장 많이 선택된 메뉴, 잘 안 뽑히는 메뉴 등) 대시보드
- 중복/비슷한 메뉴 제안 자동 병합 또는 경고 기능
