## Vibe 웹 프로젝트 AGENTS 규칙

이 프로젝트에서 AI가 코드를 작성하거나 수정할 때 따라야 할 공통 규칙이다.  
**Next.js App Router + TypeScript + Tailwind CSS + Supabase** 조합을 기본 전제로 한다.

---

## 1. 기술 스택 / 구조

- **Next.js App Router 사용**
  - `pages/` 대신 **`app/` 디렉터리** 구조를 사용한다.
  - 서버 컴포넌트 / 클라이언트 컴포넌트를 구분하고, 가능하면 서버 컴포넌트를 기본값으로 사용한다.
  - 파일 기반 라우팅 규칙(App Router)과 `layout.tsx`, `page.tsx` 구조를 따른다.

- **언어: TypeScript**
  - 모든 React 컴포넌트/유틸/서버 코드에 **`.ts` / `.tsx`** 를 사용한다.
  - 타입을 최대한 명시적으로 작성하고, `any` 사용은 꼭 필요한 예외 상황에만 허용한다.

- **스타일: Tailwind CSS**
  - 스타일링은 기본적으로 **Tailwind 유틸리티 클래스**를 사용한다.
  - 전역 CSS 파일은 리셋, 폰트, Tailwind 커스텀 설정 등 꼭 필요한 경우에만 사용한다.
  - 컴포넌트 단위로 재사용 가능한 스타일이 필요하면 **Tailwind 기반의 UI 컴포넌트**로 추출한다.

- **데이터베이스 / 백엔드: Supabase**
  - 데이터베이스, 인증, 스토리지 등은 **Supabase**를 기본으로 사용한다.
  - 가능한 한 Supabase의 공식 클라이언트와 권장 패턴을 따른다.

---

## 2. AI가 반드시 지켜야 할 것 (Do)

- **Next.js App Router 규칙 준수**
  - 페이지는 `app/(segment)/page.tsx` 형태로 생성한다.
  - 공통 레이아웃은 `app/layout.tsx` 또는 `app/(segment)/layout.tsx`에 둔다.
  - 서버 액션, 라우트 핸들러 등은 App Router 방식(`app/api/.../route.ts`)으로 작성한다.

- **TypeScript 우선**
  - 함수, 컴포넌트, 훅 등에는 **명확한 타입 정의**를 제공한다.
  - Supabase와 통신하는 코드에서는 응답 타입을 정의하거나 최소한 안전하게 다룬다(널 체크, 에러 처리 등).

- **Tailwind 스타일링**
  - 컴포넌트 스타일은 우선적으로 `className`에 Tailwind 클래스를 사용하는 방식으로 작성한다.
  - 중복되는 스타일 패턴이 생기면 **공용 UI 컴포넌트**로 추상화한다.

- **Supabase 활용**
  - 데이터 읽기/쓰기/인증 등은 별도 백엔드 서버를 만들기 전에 **Supabase 클라이언트**로 구현하는 것을 우선 고려한다.
  - 환경 변수(`process.env`)를 통해 Supabase 키/URL을 주입받는 패턴을 사용한다.

- **환경 변수 / 보안**
  - API 키, 비밀 값은 코드에 직접 쓰지 않고 **`.env` / 환경 변수**를 사용한다고 가정하고 작성한다.
  - 예시가 필요할 경우 `process.env.NEXT_PUBLIC_...` 또는 `process.env.SUPABASE_...` 형식으로만 표현한다.

## 3. AI가 하면 안 되는 것 (Don’t)

- **Next.js 구식 패턴 사용 금지**
  - `pages/` 디렉터리 기반 라우팅 예시는 만들지 않는다.
  - `getServerSideProps`, `getStaticProps` 등 **Pages Router 전용 함수 예시는 사용하지 않는다.**

- **JavaScript-only 코드 금지**
  - `.js` / `.jsx` 파일 기준의 예시는 만들지 않는다.
  - 타입을 전혀 사용하지 않는 순수 JS 스타일의 코드는 피한다.

- **Tailwind를 무시한 스타일 작성 금지**
  - 별도 이유 없이 `styled-components`, `Emotion`, `CSS Modules`, 인라인 스타일만 사용하는 패턴을 도입하지 않는다.
  - Tailwind 설정과 충돌할 수 있는 전역 CSS(리셋/폰트 외)를 무분별하게 추가하지 않는다.

- **Supabase를 우회하는 독자적인 백엔드/DB 설계 금지**
  - 프로젝트 전반의 데이터 저장을 위해 **Supabase 외에 임의의 데이터베이스(예: 직접 PostgreSQL 연결, MongoDB 등)를 기본으로 제안하지 않는다.**
  - 간단한 예제에서 로컬 배열/Mock DB를 사용하는 건 허용되지만, 실제 설계 제안 시에는 Supabase를 기준으로 설명한다.

- **비밀/키 하드코딩 금지**
  - Supabase 키, 토큰, 기타 민감 정보는 코드 안에 문자열로 직접 작성하지 않는다.
  - 예시에서도 실제 값처럼 보이는 키/토큰 문자열을 쓰지 않는다.

---

## 4. 코드 스타일 / 설계 가이드 (선호 사항)

- **컴포넌트 구조**
  - 화면 단위: `page.tsx`
  - 레이아웃/공통 UI: `layout.tsx` 또는 `components/` 디렉터리
  - 비즈니스 로직 / 데이터 접근: 별도 유틸 함수, 훅(예: `useXxx`) 또는 서버 액션으로 분리

- **에러 및 로딩**
  - 데이터 패칭이 있는 페이지/컴포넌트는 로딩/에러 상태를 고려해 설계한다.
  - App Router의 `loading.tsx`, `error.tsx`를 적극 활용하는 방향을 우선 고려한다.

- **가독성**
  - 한 파일에 너무 많은 책임을 몰아넣지 말고, UI/로직/유틸을 적절히 분리한다.
  - 함수와 컴포넌트 이름은 **의도를 드러내는 영어 이름**을 사용한다.

이 문서는 “AI에게 코드를 부탁할 때의 기본 규칙”이다.  
새로운 요구 사항이 생기면, 이 `AGENTS.md`에 규칙을 추가하거나 수정하는 방향으로 관리한다.
